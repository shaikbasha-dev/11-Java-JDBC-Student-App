# Contributing Guide

First and foremost, thank you for your interest in contributing to the **11-Java-JDBC-Student-App** repository.

This repository was created as an educational resource to help students, beginners, and interview candidates understand how Java applications communicate with relational databases using JDBC and MySQL. Every contribution that improves the quality, accuracy, or learning experience is greatly appreciated.

---

# Purpose of this Repository

The primary goals of this repository are:

- Teach JDBC fundamentals through practical examples.
- Demonstrate object persistence using PreparedStatement.
- Explain database connectivity with MySQL.
- Provide interview-oriented documentation.
- Serve as a revision resource for Java developers.
- Follow professional GitHub repository standards.

# Contribution Workflow

Follow these steps when contributing.

```text
Fork Repository
        │
        ▼
Create New Branch
        │
        ▼
Make Changes
        │
        ▼
Test Changes
        │
        ▼
Commit Changes
        │
        ▼
Push Branch
        │
        ▼
Create Pull Request
```

---

# Branch Naming Convention

Use meaningful branch names.

Examples:

```text
feature/improve-readme

feature/add-documentation

bugfix/fix-sql-query

docs/update-method-reference

refactor/improve-code
```

---

# Commit Message Guidelines

Please use clear and descriptive commit messages.

Examples:

```text
docs: improve JDBC architecture documentation

docs: add troubleshooting guide

feat: improve object insertion example

fix: correct SQL query formatting

refactor: simplify insertObject method
```

---

# Coding Guidelines

Please follow these standards.

## Java

- Use meaningful variable names.
- Follow Java naming conventions.
- Keep methods focused on a single responsibility.
- Write readable code.
- Remove unused imports.

---

## SQL

- Use parameterized queries.
- Format SQL consistently.
- Avoid unnecessary complexity.

---

## Documentation

Documentation should:

- Use proper Markdown formatting.
- Be easy to understand.
- Be technically accurate.
- Include practical examples whenever appropriate.

---

# Pull Request Checklist

Before submitting a pull request, verify the following.

- Code compiles successfully.
- Documentation has been updated if necessary.
- Formatting is consistent.
- No unnecessary files are included.
- Existing functionality remains unchanged.
- Changes have been tested.

---

# Reporting Issues

If you encounter a problem, please include:

- Clear title.
- Detailed description.
- Steps to reproduce.
- Expected behavior.
- Actual behavior.
- Screenshots (if applicable).
- Environment details.

---

# What to Avoid

Please avoid:

- Unrelated changes.
- Large formatting-only pull requests.
- Breaking existing functionality.
- Adding unnecessary dependencies.
- Copying copyrighted material.
